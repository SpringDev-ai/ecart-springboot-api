LOGIN FLOW EXPLANATION
======================

This file explains what happens when the client calls the login API:

POST /user/userLogin

Request body:
{
  "email": "rahul@gmail.com",
  "password": "Rahul@123"
}


1. CLIENT SENDS REQUEST
-----------------------

The client sends an HTTP POST request to:

/user/userLogin

with JSON body:

- email
- password

At this point the request enters the Spring Boot application.


2. REQUEST FIRST PASSES THROUGH SPRING SECURITY FILTER CHAIN
------------------------------------------------------------

Class involved:

- src/main/java/com/springboot/user_app/security/SecurityConfig.java

Code behavior:

- SecurityConfig defines a SecurityFilterChain bean.
- In this bean, the endpoint /user/register and /user/userLogin are marked as permitAll().
- Because /user/userLogin is public, Spring Security allows the request to continue without requiring Basic Authentication.

Important:

- The login endpoint itself is public.
- Spring Security does not block this request before it reaches the controller.


3. REQUEST REACHES USERCONTROLLER
---------------------------------

Class involved:

- src/main/java/com/springboot/user_app/controller/UserController.java

Method involved:

public ResponseEntity<?> userLogin(@RequestBody UserLoginRequestDto u)

Execution:

1. Spring reads the JSON request body.
2. Spring converts the JSON into a UserLoginRequestDto object.
3. The object now contains:
   - u.getEmail()
   - u.getPassword()
4. Controller calls:

   userService.userLogin(u.getEmail(), u.getPassword())

5. Controller waits for the service result.

Controller responsibility:

- Accept request
- Convert JSON to DTO
- Call service
- Return service response as HTTP response

The controller does not compare passwords itself.


4. USERLOGINREQUESTDTO HOLDS THE REQUEST DATA
---------------------------------------------

Class involved:

- src/main/java/com/springboot/user_app/dto/UserLoginRequestDto.java

This DTO only carries:

- email
- password

It acts like a transport object between client JSON and Java code.


5. CONTROLLER CALLS USERSERVICE.USERLOGIN()
-------------------------------------------

Class involved:

- src/main/java/com/springboot/user_app/service/UserService.java

Method involved:

public ResponseStructure<AuthResponseDto> userLogin(String email, String passowrd)

This is the main business method for login.

Current logic:

try {
    authenticationManager.authenticate(
        new UsernamePasswordAuthenticationToken(email, passowrd)
    );
} catch (BadCredentialsException e) {
    throw new IncorrectPasswordException();
}

User getUser = userDao.getuserByEmail(email);
AuthResponseDto authResponseDto =
    new AuthResponseDto(getUser.getEmail(), getUser.getRole(), "BASIC");

ResponseStructure<AuthResponseDto> structure = new ResponseStructure<>();
structure.setData(authResponseDto);
structure.setTime(LocalDateTime.now());
structure.setMessage("Login Success");
structure.setHttpStatusCode(200);
return structure;


6. AUTHENTICATIONMANAGER STARTS LOGIN VALIDATION
------------------------------------------------

The first important line is:

authenticationManager.authenticate(
    new UsernamePasswordAuthenticationToken(email, passowrd)
);

What this means:

- A UsernamePasswordAuthenticationToken is created from the email and password sent by the client.
- That token is handed to Spring Security's AuthenticationManager.
- AuthenticationManager now tries to validate the credentials.

This is better than manual string comparison because Spring Security handles the authentication flow in a standard way.


7. SPRING SECURITY USES APPLICATIONUSERDETAILSSERVICE
-----------------------------------------------------

Class involved:

- src/main/java/com/springboot/user_app/security/ApplicationUserDetailsService.java

Method involved:

public UserDetails loadUserByUsername(String username)

Execution:

1. Spring Security receives the username.
2. Here username is actually the email.
3. Spring calls:

   userRepository.findByEmail(username)

4. If user is not found, UsernameNotFoundException is thrown.
5. If user is found, Spring builds a Spring Security UserDetails object using:
   - stored email
   - stored encoded password
   - role converted to authority like ROLE_USER or ROLE_ADMIN

Returned object:

User.withUsername(user.getEmail())
    .password(user.getPassword())
    .authorities(new SimpleGrantedAuthority("ROLE_" + role.name()))
    .build();

So this layer converts your database user into a format Spring Security understands.


8. USERREPOSITORY READS THE USER FROM DATABASE
----------------------------------------------

Class involved:

- src/main/java/com/springboot/user_app/repository/UserRepository.java

Method involved:

Optional<User> findByEmail(String email)

Execution:

1. Spring Data JPA generates the SQL automatically.
2. It queries the database using the email.
3. It returns Optional<User>.

This is the actual database access point used during authentication.


9. PASSWORD CHECK HAPPENS INSIDE SPRING SECURITY
------------------------------------------------

Class involved indirectly:

- PasswordEncoder bean from SecurityConfig
- BCryptPasswordEncoder

During authentication:

1. Spring gets the plain password from the request.
2. Spring gets the encoded password from the database.
3. BCryptPasswordEncoder compares them using matches().

Example idea:

- request password: Rahul@123
- database password: $2a$10$.....

Spring does not decode the stored password.
Instead it hashes the incoming password in a compatible way and checks if both match.

If matched:

- authentication succeeds

If not matched:

- BadCredentialsException is thrown


10. FAILED LOGIN PATH
---------------------

If credentials are wrong:

1. AuthenticationManager throws BadCredentialsException
2. UserService catches it
3. UserService throws IncorrectPasswordException
4. UserExceptionController handles it
5. A structured error response is returned

Class involved:

- src/main/java/com/springboot/user_app/Exception/UserExceptionController.java

Response structure for failure:

- data: "Incorrect Password"
- message: "Incorrect Password"
- time: current time
- httpStatusCode: 401

HTTP status returned:

- 401 Unauthorized


11. SUCCESS PATH CONTINUES AFTER AUTHENTICATION
-----------------------------------------------

If authentication succeeds:

1. Control returns to UserService.userLogin()
2. Then service calls:

   User getUser = userDao.getuserByEmail(email);

Why call database again?

- Authentication already verified the credentials.
- This second fetch is used to build the application response with email and role.


12. USERDAO FETCHES USER AGAIN
------------------------------

Class involved:

- src/main/java/com/springboot/user_app/dao/UserDao.java

Method involved:

public User getuserByEmail(String mail)

Execution:

1. UserDao calls userRepo.findByEmail(mail)
2. If present, user object is returned
3. If absent, EmailNotFoundException is thrown

This DAO layer is your application's wrapper over the repository.


13. SERVICE CREATES AUTHRESPONSEDTO
-----------------------------------

Class involved:

- src/main/java/com/springboot/user_app/dto/AuthResponseDto.java

Object created:

new AuthResponseDto(getUser.getEmail(), getUser.getRole(), "BASIC")

Meaning of fields:

- email: logged-in user's email
- role: USER or ADMIN
- authType: BASIC

Why this DTO is useful:

- It avoids returning the full User entity
- It does not expose password
- It sends only login-related response data


14. SERVICE WRAPS DTO IN RESPONSESTRUCTURE
------------------------------------------

Class involved:

- src/main/java/com/springboot/user_app/response/ResponseStructure.java

The service sets:

- data = AuthResponseDto
- time = current LocalDateTime
- message = "Login Success"
- httpStatusCode = 200

So the service produces one standard response object for the controller.


15. CONTROL RETURNS TO USERCONTROLLER
-------------------------------------

Back in UserController:

ResponseStructure<?> structure = userService.userLogin(u.getEmail(), u.getPassword());
return new ResponseEntity<>(structure, HttpStatus.OK);

Execution:

1. Controller receives the ResponseStructure from service
2. Controller wraps it in ResponseEntity
3. HTTP status 200 OK is sent back to the client


16. FINAL CLIENT RESPONSE
-------------------------

Example successful response:

{
  "data": {
    "email": "rahul@gmail.com",
    "role": "USER",
    "authType": "BASIC"
  },
  "time": "2026-04-19T10:00:00",
  "message": "Login Success",
  "httpStatusCode": 200
}

This is the final JSON sent back to the client.


17. COMPLETE LAYER-BY-LAYER CHAIN
---------------------------------

Full execution chain for successful login:

1. Client
2. Spring Security Filter Chain
3. UserController.userLogin()
4. UserService.userLogin()
5. AuthenticationManager.authenticate()
6. ApplicationUserDetailsService.loadUserByUsername()
7. UserRepository.findByEmail()
8. BCryptPasswordEncoder password comparison
9. Back to UserService
10. UserDao.getuserByEmail()
11. UserRepository.findByEmail()
12. AuthResponseDto creation
13. ResponseStructure creation
14. UserController returns ResponseEntity
15. Client receives JSON response


18. SIMPLE MENTAL MODEL
-----------------------

You can remember the login flow like this:

- Controller receives request
- Service starts authentication
- Spring Security checks credentials
- Repository reads user from DB
- BCrypt verifies password
- Service builds safe response DTO
- Controller returns JSON to client


19. IMPORTANT NOTE FOR PROTECTED APIS
-------------------------------------

This login endpoint only validates credentials and returns a success response.

It does NOT create:

- JWT token
- session id
- refresh token

For protected APIs, the client must still send Basic Auth on each request.

Example:

Authorization: Basic base64(email:password)
